# Architecture

Agent Isles has three layers.

## 1. Source format

The source format is standard Markdown with explicit HTML islands.

Principles:

- Prose, headings, lists, tables, and code fences stay as Markdown.
- Rich repeated patterns use semantic custom elements.
- The source remains readable even if a viewer ignores the custom elements.

Example:

```markdown
# Deployment Decision

<agent-decision verdict="ship-with-guardrails" title="Proceed">
Deploy after CI passes and rollback notes are confirmed.
</agent-decision>
```

## 2. Component vocabulary

The initial vocabulary uses Lit Web Components under the `<agent-*>` prefix. The durable reference is [Component Vocabulary](Component-Vocabulary) in the wiki mirror and `docs/component-vocabulary.md` in the repo.

Implemented now:

- `<agent-decision>`
- `<agent-risk>`

Planned next:

- `<agent-metric>`
- `<agent-copy-block>`
- `<agent-tabs>` and `<agent-tab>`
- `<agent-timeline>` and `<agent-step>`

The component prefix should remain boring and explicit. The island metaphor belongs in the product brand, not necessarily every API name.

## 3. CLI renderer

The renderer is a Node ESM pipeline:

```txt
Markdown file
  -> unified
  -> remark-parse
  -> remark-gfm
  -> remark-rehype with raw HTML enabled
  -> rehype-raw
  -> rehype-highlight
  -> rehype-stringify
  -> full HTML document
```

The HTML page injects:

- Bootstrap CSS and JS
- highlight.js CSS
- Agent Isles theme CSS
- `agent-components.js` generated by Rollup

## Security boundary

The current renderer supports trusted Markdown with raw HTML islands. That is useful for agent-authored files in a controlled repo, but it is not safe for arbitrary untrusted input.

Issue [#6](https://github.com/zpyoung/agent-isles/issues/6) tracks adding a sanitized mode.
